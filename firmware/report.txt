1. Problem Statement
Firmware updates are a critical maintenance operation for embedded IoT devices deployed in the field.
However, a failed or interrupted update — caused by power loss, network disruption, hardware fault, or
corrupted binaries — can permanently brick a device, making it non-recoverable without physical
intervention. This is especially severe in large-scale IoT deployments where physical access is impractical.
The challenge is to design a Fail-Safe Over-The-Air (OTA) Firmware Update System for an ESP32
microcontroller that guarantees the device always remains in a known-good, operational state regardless
of what goes wrong during the update process. The system must handle partial downloads, corrupted
binaries, post-update boot failures, and network instability — all without human intervention.

2. Detailed Architecture
The system is organized into three layers: the Cloud/Server Layer, the Network Transport Layer, and the
ESP32 Device Layer. Each layer has a clear responsibility, and failure in any one layer is handled
gracefully without affecting the others.

Figure 1 — Full System Architecture Block Diagram

2.1 Layer Breakdown
■ Cloud / Server Layer: FastAPI Python server manages firmware versions, stores .bin files, and
triggers updates via MQTT publish. A Rust CLI tool pre-processes each firmware binary to generate
SHA-256 hash and metadata JSON before publishing.
■ Network Layer: MQTT over TLS 1.3 via a self-hosted Mosquitto broker. QoS Level 1 guarantees
at-least-once delivery. Firmware is split into 4KB chunks — each chunk carries a sequence number
and CRC32 checksum.
■ ESP32 Device Layer: An OTA Agent FreeRTOS task handles download, verification, and the
write to Partition B. The ESP32 native bootloader manages the A/B partition swap and rollback logic at
the hardware level.

3. Detailed Methodology
3.1 A/B Dual Partition Strategy
The ESP32 flash is divided into two equal firmware partitions — Partition A (active) and Partition B
(staging). The device always boots from the partition marked active in the OTA Data Partition. New
firmware is written exclusively to the staging partition, ensuring the running firmware is never touched
during an update. This is the foundational safety guarantee.

Figure 2 — ESP32 Flash Memory Layout with A/B Partitioning

3.2 The 3-Stage Commit Protocol
The update lifecycle follows a strict 3-stage protocol, leveraging the ESP32 bootloader's native state
machine. This is not custom rollback logic — it is hardware-guaranteed.

Stage 1 — Download & Verify
The OTA Agent subscribes to MQTT firmware topics. Chunks arrive sequentially; each is validated with
CRC32 before being written to Partition B. After all chunks, a full SHA-256 hash of the written binary is
compared against the server-published hash.

3.4 Boot Loop Detection
On every boot, the firmware increments a reboot counter stored in NVS. After a successful self-test and
commit, the counter is reset to 0. If the counter reaches 3 without a commit, the bootloader automatically
rolls back to the previously committed partition. This handles scenarios where the device crashes during
the self-test window before the counter can be reset.

4. MQTT Communication Protocol
MQTT is chosen for its lightweight pub/sub model, ideal for constrained embedded devices. The broker
(Mosquitto) acts as the decoupling layer between the backend server and the device fleet. TLS 1.3
secures all communication. QoS Level 1 ensures firmware chunks are delivered at least once, with the
device-side deduplication using sequence numbers.

7. SanDisk / Flash Memory Relevance
This project directly addresses core challenges in flash memory management — central to SanDisk's
domain of expertise.

■ Write Endurance Awareness
OTA writes are among the heaviest flash operations. Our A/B design concentrates writes to Partition B
during updates, protecting Partition A (active) from unnecessary wear cycles.

■ Atomic Write Guarantees
Flash memory does not support true atomic multi-block writes. Our system writes only to the staging
partition and only commits by updating a single OTA Data flag — mimicking atomic semantics over NOR
flash.

■ NVS as Reliable Metadata Store
The NVS (Non-Volatile Storage) library implements its own wear leveling over the flash, making it safe to
store frequently-updated counters like the reboot counter.

■ NAND vs NOR Considerations
ESP32 uses NOR flash — byte-addressable and suitable for XIP (execute-in-place). Our design
acknowledges this: Partition A runs XIP while Partition B is written page-by-page.

■ Power-Loss Resilience
Unlike storage devices that buffer writes, embedded NOR flash writes are immediate but non-atomic
across pages. Our CRC32 per-chunk and full SHA-256 detect any partial page corruption from power loss
during download.
